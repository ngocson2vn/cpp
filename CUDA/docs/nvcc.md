# Compiler Options
## Generate PTX and CUBIN
```bash
nvcc ... -gencode arch=compute_100,code=sm_100
```
To understand the difference between `arch` and `code` in the `nvcc` compiler, you have to look at how CUDA code is compiled. NVIDIA uses a **two-stage compilation process**: first to an intermediate virtual format, and then to a hardware-specific binary format.

Here is the breakdown of how `arch` and `code` control these two stages.

### 1. `arch`: The Virtual Architecture (The Baseline)

The `arch` argument specifies the **compute capability of a virtual GPU**. It dictates the first stage of compilation.

* **What it generates:** PTX (Parallel Thread Execution). This is an intermediate, assembly-like language created by NVIDIA. It is not tied to the physical wiring of any specific GPU.
* **What it does:** It sets the "baseline" of hardware features your code is allowed to use. For example, if you set `arch=compute_80`, you are telling the compiler: *"Assume the GPU running this has at least the features of compute capability 80 (Ampere), like third-generation Tensor Cores."* * **The Catch:** A physical GPU cannot run PTX directly. It must be translated into hardware-specific binary code (SASS) before it can execute.

### 2. `code`: The Real Architecture (The Payload)

The `code` argument specifies **what actually gets embedded into your final executable file**. It dictates the second stage of compilation.

It can take two different types of values, which serve completely different purposes:

* **Hardware-Specific Binary (`code=sm_XX`):** This takes the PTX generated by `arch` and compiles it all the way down to a highly optimized, machine-level binary (SASS/`cubin`) for a specific physical GPU.
* *Result:* The executable will run instantly on an `sm_XX` GPU, but it **will fail** on newer or older architectures because the binary wiring doesn't match.


* **Intermediate Code for JIT (`code=compute_XX`):** This tells the compiler to skip generating a hardware binary and instead embed the raw, intermediate PTX code directly into the executable.
* *Result:* When you run the program, the NVIDIA graphics driver on the host machine will grab that PTX and perform Just-In-Time (JIT) compilation to turn it into the correct binary for whatever GPU happens to be installed. This guarantees forward compatibility with future GPUs, but adds a slight delay the very first time the application launches.

# Errors
## Error: undefined hidden symbol
Even though the function template `void lego_ops::softmax_top1_kernel<float>(float const*, int*, float*, int, int)` was manually instantiated, <br/>
the following error still occurs when compiling with CUDA 13.x:
```bash
ld.lld: error: undefined hidden symbol: void lego_ops::softmax_top1_kernel<float>(float const*, int*, float*, int, int)
>>> referenced by tmpxft_0028743a_00000000-6_moe.compute_100.cudafe1.cpp
```
This error is caused by a **breaking change introduced in CUDA 13.0** (and carried over to 13.1.0) regarding the default symbol visibility of device kernels.

### The Cause

In previous versions of CUDA, `__global__` functions (kernels) had default visibility, meaning they were exported and visible across shared library boundaries.

**Starting with CUDA 13.0**, the compiler (NVCC) defaults `__global__` functions to **hidden visibility** to prevent symbol collisions and improve safety in complex builds. This means that even if you manually instantiate your template:

1. The compiler generates the code for `softmax_top1_kernel<float>`.
2. It marks the symbol as `hidden` (private to that translation unit or shared library).
3. When another object file (like the generated `tmpxft_...` intermediate file) tries to link against it, the linker (`ld.lld`) rejects it because it cannot "see" the hidden symbol, resulting in `undefined hidden symbol`.

### Solutions

You can fix this in one of two ways:

#### Option 1: Revert to Legacy Behavior (Recommended for Quick Fix)

Pass the following flag to `nvcc` to disable the new hidden visibility behavior. This restores the behavior found in CUDA 12.x and earlier.

```bash
nvcc ... --device-entity-has-hidden-visibility=false ...

```

If you are using CMake, add this to your `CMAKE_CUDA_FLAGS`:

```cmake
set(CMAKE_CUDA_FLAGS "${CMAKE_CUDA_FLAGS} --device-entity-has-hidden-visibility=false")

```

#### Option 2: Explicitly Mark the Template as Visible

If you prefer to keep the new stricter defaults, you must explicitly mark your specific kernel template as visible in its definition/declaration.

Modify your code for `lego_ops::softmax_top1_kernel`:

```cpp
// Add __attribute__((visibility("default")))
template <typename T>
__global__ __attribute__((visibility("default")))
void softmax_top1_kernel(const T* input, int* output, float* val, int rows, int cols) {
    // ... kernel implementation ...
}

```

#### Option 3: Check `rdc` (Relocatable Device Code)

If you are compiling with Relocatable Device Code enabled (`-rdc=true`), you may also need to ensure that the template stub behavior matches your linkage expectations by adding:

```bash
-static-global-template-stub=false

```

### Summary of Flags

| Flag | Purpose |
| --- | --- |
| `--device-entity-has-hidden-visibility=false` | **Fixes your error.** Forces kernels to be visible (public) like in CUDA 12. |
| `-static-global-template-stub=false` | Prevents the compiler from making template stubs internal (static). |


## Error: no kernel image is available for execution on the device
When you compile with `-gencode arch=compute_100,code=sm_100`:

* `arch=compute_100`: Instructs the compiler to generate intermediate virtual architecture code (PTX) for compute capability 100.
* `code=sm_100`: Instructs the compiler to take that PTX, compile it down to a hardware-specific binary (`cubin`) exclusively for `sm_100`, and embed **only** that binary into the final executable.

Because `cubin` files are strictly tied to specific hardware architectures, an `sm_100` binary cannot run on an `sm_120` GPU. When the CUDA runtime launches the kernel, it searches the executable for an `sm_120` binary or forward-compatible PTX. Finding neither, it throws the `no kernel image is available for execution on the device` error.


If you know you are going to run the code on an `sm_120` device, you should add a generation flag specifically for it. This avoids the overhead of JIT compilation when the application starts.

Update your compile command to include both:

```bash
nvcc ... -gencode arch=compute_100,code=sm_100 \
         -gencode arch=compute_120,code=sm_120
```

If you want your executable to be forward-compatible with any future GPU architecture (like `sm_120` and beyond), you need to tell the compiler to embed the PTX intermediate code.

You do this by adding `compute_100` to the `code=` argument:

```bash
nvcc ... -gencode arch=compute_100,code=\"sm_100,compute_100\"
```

*Note the escaped quotes around the `code` arguments.*

**What this does:**

1. Embeds the `sm_100` binary (so `sm_100` devices can run it instantly).
2. Embeds the `compute_100` PTX code.
3. When you launch the program on your `sm_120` device, the NVIDIA driver sees the PTX code and automatically JIT-compiles it into an `sm_120` binary on the fly.
